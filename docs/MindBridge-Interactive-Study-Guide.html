Ideas & vocabulary (beginner)

Repository (repo)

A folder of files tracked with version control (here: Git). The MindBridge repo contains frontend/, backend/, and docs.

User interface (UI)

What people see and click in the browser: buttons, forms, pages. Built with React in frontend/.

Server / backend

A program that runs on a machine and answers HTTP requests. Here it is Django in backend/. It talks to PostgreSQL (a database) to store moods, users, appointments.

API (Application Programming Interface)

A set of URLs the frontend calls to read or write data. Example: “GET my moods” returns a list. MindBridge uses JSON (text that looks like structured data) in request and response bodies.

Authentication & JWT

After login, the server returns tokens (long secret strings). The frontend sends an access token on each request. When it expires, the refresh token can be traded for a new access token (see frontend/src/lib/api.ts).

Role-based access

Users have a role: patient, counsellor, or admin (stored on Profile in the database). The UI and the API both restrict what each role can do.

Routing

In the browser, the URL path (like /mood) picks which React page to show. In Django, URL patterns map paths to Python views.

Model / migration

A model is a Python class describing a database table. A migration is a recorded change to the database schema (applied with Django commands).

Rasa

A separate conversational AI service. Django’s chatbot app forwards messages to Rasa’s HTTP webhook.

Suggested study path

  1. frontend/src/main.tsx → App.tsx → AuthContext.tsx (how login state works).
  2. frontend/src/lib/api.ts (every network call in one place).
  3. Pick one screen: e.g. MoodTracker.tsx and trace it to moods/views.py.
  4. Read backend/mindbridge_api/urls.py then each app’s urls.py.
  5. Read appointments/views.py + permissions.py for role checks.

Frontend deep dive

The frontend is a single-page application: one HTML shell loads JavaScript that swaps “pages” without full reloads. Vite bundles TypeScript during development.

Entry & routing

  • main.tsx mounts React into #root, wraps BrowserRouter and AuthProvider.
  • App.tsx shows Login / SignUp if not authenticated; otherwise Routes inside MainLayout.

Auth

AuthContext holds tokens and me (user profile). Tokens persist in localStorage under fixed keys.

API layer

api.ts exports async functions that call Django. Many repeat the pattern: try request → if 401, refresh token → retry once.

Backend deep dive

Django organizes code into apps (folders like moods/). Django REST Framework turns Python classes into JSON APIs. Simple JWT handles login tokens.

Request lifecycle (simplified)

  1. Browser sends HTTP request to e.g. /api/moods/.
  2. mindbridge_api/urls.py includes moods.urls.
  3. A view runs, checks permissions, reads/writes models via serializers.
  4. Response JSON goes back to the browser.

Rasa chatbot

Training data lives under backend/rasa/data/. domain.yml lists intents, responses, and actions. Custom Python behavior is in actions/actions.py (classes extending Rasa SDK Action). Django posts to Rasa’s REST webhook; see chatbot/views.py chatbot_message.

API map (Django)

Base URL is typically http://127.0.0.1:8000 in development. Prefix /api/ as below.

Method & pathPurpose
POST /api/auth/register/Create user (AllowAny).
POST /api/auth/login/JWT access + refresh tokens.
POST /api/auth/refresh/New access token from refresh.
GET /api/auth/me/Current user + role (authenticated).
GET /api/auth/counsellors/Verified counsellors list.
GET,POST /api/moods/List or create moods for current user.
POST /api/appointments/create/Patient creates appointment.
GET /api/appointments/my/Patient’s appointments.
GET /api/appointments/counsellor/Counsellor’s appointments.
PATCH /api/appointments/<id>/Counsellor updates status.
GET /api/appointments/available/<counsellor_id>/?date=YYYY-MM-DDFree hourly slots.
POST /api/chatbot/message/Proxy to Rasa (AllowAny in code).

Note: Backend permission IsCounsellor only allows role counsellor. The frontend treats admin like a counsellor for navigation; if admins need API access, the permission class would need updating.

Self-check quiz

1. What file centralizes HTTP calls from React to Django?

Answer: frontend/src/lib/api.ts

2. Where is the user’s role (patient/counsellor/admin) stored in the database?

Answer: On the Profile model linked one-to-one to Django’s User (backend/accounts/models.py).

3. Which view builds the list of available appointment hours?

Answer: AvailableSlotsView.get in backend/appointments/views.py.

4. Why does the app send an Authorization: Bearer … header?

Answer: To prove the request comes from a logged-in user (JWT access token).

5. Are wellness resources on the Resources page loaded from PostgreSQL?

Answer: No — they come from the static array in frontend/src/data/resourcesData.ts. Django Resource models exist for future use.
